# 0010 ‚Äî Implementation defaults (correlation_id, event_number, canonicalization, locale, email scope, value sets, audit log, profile validation, license)

**Status:** Accepted

## Context

The readiness review surfaced nine engineering decisions that the design either glossed over or split across multiple documents. They are not architectural ‚Äî every one is a routine project-default question ‚Äî but coding cannot start without them pinned because multiple components depend on the same answer and an implementer guessing differently in two places creates a contract break.

This ADR pins each one. None of these is FHIR-spec-mandated; all are project defaults bounded by the architecture and earlier ADRs.

## Decisions

### 1. `correlation_id` is UUIDv4, stored as Postgres `uuid`

The `correlation_id` field on `hl7_message_queue`, `resource_changes`, `ehr_events`, `deliveries`, `pending_pairs`, and the `AdapterContext` / `ResourceChange` SPI types is a UUIDv4. Generated by the originating component (the MLLP listener for HL7 messages, the FHIR scan runner for scan results, the vendor API client for change-feed records). Stored as a Postgres `uuid` column end-to-end ‚Äî no `text` storage, no string serialization between stages.

**Why.** Random, fixed-width, ubiquitous Go support (`github.com/google/uuid`), index-friendly, no inter-component coordination needed, no cross-deployment collision risk. ULID was the only serious alternative (lexicographic sort) but its sort-order benefit is subsumed by `event_number` and `sequence` which are already monotonic.

**Wire form.** When a `correlation_id` crosses a network boundary (logs, traces, audit, FHIR resource extensions), it is the canonical 36-character lowercase hyphenated form (`xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx`).

### 2. `event_number` is deployment-wide on `ehr_events` and per-subscription on `deliveries`

Two columns, two scopes, not the same number on each table:

- **`ehr_events.event_number`** ‚Äî `bigint`, server-assigned monotonic, deployment-wide. Backed by a single Postgres `SEQUENCE` (`ehr_events_event_number_seq`). The Topic Matcher calls `nextval` inside the transaction that inserts the row. Drives `$events` ordering and operator-side debugging.
- **`deliveries.event_number`** ‚Äî `bigint`, per-subscription monotonic, assigned at fanout (Stage 3). Computed as `max(deliveries.event_number) + 1 WHERE subscription_id = ?`. Drives `Subscription.eventsSinceSubscriptionStart` and is what subscribers see in `notificationEvent.eventNumber`. Renumbering at fanout is what makes per-subscriber ordering correct regardless of how many topics matched the same change.

Both columns participate in unique constraints: `UNIQUE (event_number)` on `ehr_events` (implicit via the sequence), `UNIQUE (subscription_id, event_number)` on `deliveries`.

**Why.** The spec wants per-subscription monotonic at the wire (`eventsSinceSubscriptionStart`), but operators and `$events` replay want a deployment-wide sequence for ordering arbitrary slices of history. A single shared sequence cannot satisfy both. Two columns, two scopes, no name overload.

### 3. JSON canonicalization uses RFC 8785 (JCS)

[RFC 8785 ‚Äî JSON Canonicalization Scheme (JCS)](https://www.rfc-editor.org/rfc/rfc8785) is the canonicalization rule wherever this project hashes or signs JSON:

- **FHIR Scan Runner** ‚Äî `content_hash` of a scanned resource (with volatile fields stripped first, then JCS-canonicalized, then SHA-256). Deterministic across pretty-print, key-order, whitespace, and trailing-zero floating-point variations.
- **Audit log** ‚Äî every audit row's `prev_hash` field is `SHA-256(JCS(prior_row.canonical_form))`. The chain re-walk uses the same canonicalization.
- **ETag** for `CapabilityStatement`, `SubscriptionTopic`, and other read-mostly resources ‚Äî `"<base64url(SHA-256(JCS(body)))>"`.

**Why.** Three call sites must agree byte-for-byte or the audit chain breaks and the ETag flips on a re-serialize. RFC 8785 is the only widely-implemented JSON canonicalization standard; alternatives (FHIR canonical JSON for digital signatures, project-private rules) either narrow scope or risk silent incompatibility with future tooling.

**Library.** Go implementation: `github.com/cyberphone/json-canonicalization/go/src/webpki.org/jsoncanonicalizer` or a vetted fork. If we hand-roll, we conform exactly to the RFC and ship golden-file conformance tests against the RFC's published test vectors.

### 4. Case/accent folding uses ICU root locale

Wherever the design says "case-insensitive" or "accent-insensitive" matching (notably `topic-matcher`'s string filter parameter with `:contains` modifier, FHIR search-parameter token matching where the spec calls for it, and any future text-normalization in adapter-supplied predicates), we fold using **ICU root locale** rules:

- Case folding: ICU root `cf` (case-fold).
- Diacritic stripping: NFD decomposition + remove combining marks (Unicode `\p{Mn}`).
- Order: NFC ‚Üí fold ‚Üí strip ‚Üí NFC.

**Why.** Spec says "case-insensitive" without pinning a locale. Turkish ƒ± / German √ü / French √© cases differ across language-specific implementations. ICU root is what HL7's reference implementations and major FHIR servers use; sticking with it avoids per-locale surprise.

**Library.** Go: `golang.org/x/text/cases` for case folding, `golang.org/x/text/unicode/norm` for normalization, `golang.org/x/text/runes` for diacritic stripping.

### 5. Email channel ships SMTP-only for v1; S/MIME and Direct deferred to v2

The `email` channel in v1 supports **plain SMTP / SMTPS** delivery to a configured relay (the operator's MTA, AWS SES, SendGrid, etc.). `S/MIME` and `Direct` modes ‚Äî including the certificate handling, trust-bundle refresh, and HISP integration described in [low-level-design/channels.md](../../low-level-design/channels.md) ‚Äî are **not part of v1**.

Operationally:
- `channels.email.mode` config field accepts only `"smtp"` in v1. `"smime"` and `"direct"` are documented but cause a startup error if configured.
- The Channel SPI's email module ships with the SMTP submission path complete, MIME assembly with attachment-vs-inline thresholds, SMTP AUTH, STARTTLS, and the standard `DeliveryOutcome` mapping (4xx ‚Üí TransientFailure, 5xx ‚Üí PermanentFailure).
- `S/MIME` and `Direct` will land as a v2 feature once a Go S/MIME library has been evaluated and either picked or replaced with a hand-rolled subset. Tracked separately.

**Why.** The Go S/MIME ecosystem is the weakest of the dependency surfaces this project needs. Forcing it into v1 risks either shipping with an unmaintained library or hand-rolling a CMS/PKCS#7 implementation that nobody on the team has authored before. The email channel is opt-in and most subscribers will use rest-hook; deferring is honest about scope.

### 6. ValueSets live in a mounted directory of FHIR ValueSet JSON files

The FHIR Subscriptions spec allows the `:in` modifier on filter parameters with a value set as the right-hand side. Where we support `:in` (per the search-parameter subset documented in [low-level-design/topic-matcher.md](../../low-level-design/topic-matcher.md)), the right-hand value sets are sourced from a configured directory:

- Config field: `topics.value_sets_dir` (a path; default `/etc/fhir-subs/value-sets`).
- Files: one FHIR `ValueSet` JSON resource per file. Filename is informational; the canonical URL inside the resource is the identity.
- Loaded at startup and reloaded on SIGHUP (same trigger that reloads the topic catalog).
- Pre-expanded at load time. Value sets that fail to parse or fail to expand are rejected at load with an operator-visible error (same model as topic catalog).
- Topics referencing a value set by canonical URL are validated against the loaded set. Missing references reject the topic at compile.

**Why.** Same operational shape as the topic catalog. No external terminology server in v1 ‚Äî that adds an availability dependency we do not want at this stage. Operators that need terminology-server-backed value sets can add it in v2 by registering an alternate loader.

### 7. Audit log is a durable Postgres table plus a configurable real-time sink

`audit_log` is **both** a Postgres table (the durable, hash-chained source of truth) **and** a real-time sink the operator chooses for live forwarding. Per-row writes happen to both; the table write is transactional and the sink write is best-effort (lost sink writes are tolerable because the table is the truth).

- **Table** ‚Äî defined in the v0 schema migration. Append-only, hash-chained per [low-level-design/observability.md](../../low-level-design/observability.md). Every audit-emitting component writes a row inside the same transaction as the operation it's auditing.
- **Sink** ‚Äî `observability.audit_log.sink` config field, default `stdout`. Other accepted values: `file`, `syslog`, `otlp`, `none`. The sink reads committed rows from the table and forwards. A stalled sink does not block writes.
- **Hash chain** ‚Äî every row carries `prev_hash = SHA-256(JCS(prior_row.canonical_form))`. A daily verifier walks the chain; mismatches fire a `fhir_subs_audit_chain_invalid_total` alert.

This **supersedes** the inconsistency between [decisions/0008](0008-resolved-design-questions.md#11) (which said "stdout default for the sink") and `internal-tables.md` (which lists `audit_log` as a table). Both are correct: the table is durable storage, the sink is a forwarding surface.

**Why.** Durability with operator-friendly real-time export is the pattern used by every healthcare audit-log system in production. Either alone is insufficient: stdout-only loses on container restart, table-only forces operators to run their own SIEM tail.

### 8. FHIR profile validation in v1: base R5 only, with a swap hook

The HL7 Message Processor's `validate` step and the Subscriptions API's input validation use the **base R5 FHIR profiles only** in v1. Operators who need stricter facility-specific or vendor-specific profiles will configure them in v2.

- Base profiles for the resource types we touch (`Subscription`, `SubscriptionTopic`, `SubscriptionStatus`, `Bundle`, `MessageHeader`, plus the resource types built-in topics fire on: `ServiceRequest`, `Observation`, `DiagnosticReport`, `Encounter`, `DocumentReference`, `AllergyIntolerance`, `MedicationRequest`, `MedicationStatement`, `Patient`, `Practitioner`).
- Validation library: `github.com/google/fhir` Go bindings if they cover R5 with our profile set; otherwise a hand-rolled minimum-viable validator that checks cardinality, type, value-set bindings, and required-field presence per the StructureDefinitions.
- A `ProfileValidator` interface lets operators swap in a stricter validator at startup (config field `validation.profile_validator: "base" | "us-core" | "<custom>"`). Custom validators implement the interface and are registered at compile time, mirroring the Adapter SPI loading model.

**Why.** Base R5 is what the project's wire shape requires for spec compliance. US Core, IPS, and facility-specific profiles add validation surface that operators may or may not want. Picking one for v1 forces a choice that doesn't belong in the core; punting it to a swappable interface lets operators decide without forcing everyone to install the validation pack.

### 9. License: Apache 2.0

The project ships under the [Apache License, Version 2.0](https://www.apache.org/licenses/LICENSE-2.0). Every file gets the standard header. The repo's `LICENSE` file is the canonical text.

**Why.** The standard for FOSS healthcare integration projects (HAPI FHIR, OpenMRS, OpenEMR all ship Apache 2.0 or compatible). Permissive enough for vendor adoption, includes the patent grant, compatible with downstream commercial use, and known by every contributor's legal team.

## Consequences

**Positive.**
- Implementer guidance is unambiguous on every previously-flagged item. Two consumers cannot disagree about contract shape because the contract is pinned at one place per question.
- The v0 schema can now be authored ‚Äî the column types (`uuid` for correlation, `bigint` for sequences, etc.) are concrete.
- The repo can be initialized with the locked Go library set, the Apache 2.0 license, and a coherent `audit_log` table-plus-sink shape.

**Negative.**
- Email v1 ships without S/MIME or Direct. Healthcare operators expecting Direct interoperability will not get it until v2. Documented up front.
- ValueSet pre-load means operators with millions of codes in a value set (rare, but possible for some ICD-10/SNOMED expansions) need a memory-aware load strategy. Acceptable trade-off vs. depending on a terminology server in v1.
- ICU root locale folding is not what every spec implementor uses. Operators who notice a Turkish-ƒ± edge case will need a configurable locale knob; not in v1.
- Hand-rolled FHIR profile validator may diverge from HAPI's interpretation of edge cases. The conformance suite (Inferno, Touchstone) is the reconciliation point.

**Neutral.**
- Apache 2.0 is the obvious license; no real downside to discuss.
- RFC 8785 is the only viable JSON canonicalization choice; alternatives are weaker.
