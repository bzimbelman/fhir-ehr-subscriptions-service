## BLOCKER findings — narrative

### Finding 158 — Migration runner has a fresh-DB race window

**Tag:** RACE **Severity:** BLOCKER
**Path:** `internal/infra/storage/migrate/migrations/0001_init.sql:16,302`, `internal/infra/storage/migrate/migrate.go:195-203`

```sql
create table schema_migrations (    -- line 16, no IF NOT EXISTS
    version text primary key,
    applied_at timestamptz not null default now()
);
...
insert into schema_migrations (version) values ('0001');  -- line 302, no ON CONFLICT
```

`migrate.go:195`:
```go
exists, err := schemaMigrationsExists(ctx, conn.Conn())
```
The check happens BEFORE the per-migration loop. With two pods racing on a fresh DB:

- Pod A acquires `pg_advisory_lock`, probes `exists=false`, applies 0001 (creates table, inserts ('0001')), commits, releases lock.
- Pod B was queued on `pg_advisory_lock`, finally acquires, probes `exists=...` — but pod B's `schemaMigrationsExists` may have run BEFORE the lock acquisition (the variable is captured before the loop body). Implementation actually probes after lock acquisition.

Rechecking implementation: line 195 is `exists, err := schemaMigrationsExists(...)` — yes, AFTER `pg_advisory_lock`. So pod B sees `exists=true` and skips re-creating the table. Cleared.

But: pod B's `applied[version]` map is empty (because pod B's snapshot of `schema_migrations` was taken AFTER pod A inserted '0001'). So pod B reads `applied = {"0001": ""}`, sees `m.Version == "0001"`, sees `present=true`, and continues to migration 0002. OK.

Actual issue: migration 0001's `insert into schema_migrations (version) values ('0001');` (line 302) has NO `ON CONFLICT`. If pod A's `withTx` commits the body but the runner-side `INSERT INTO schema_migrations(version, applied_at, checksum)` (migrate.go:275-278) follows in the same transaction — wait, that's the runner's INSERT not the migration's. Line 302 of the SQL body itself does `insert into schema_migrations (version) values ('0001')` — and the runner ALSO inserts via `INSERT INTO schema_migrations(version, applied_at, checksum) VALUES ($1, now(), $2)` at migrate.go:275. The migration's body inserts version '0001' into `schema_migrations`, then the runner's wrapper code (within the SAME transaction) ALSO inserts '0001'. Result: primary key violation `duplicate key value violates unique constraint "schema_migrations_pkey"`. The transaction aborts. Migration 0001 fails on EVERY fresh DB. Cleared on closer inspection: `INSERT ... ON CONFLICT (version) DO UPDATE SET checksum = excluded.checksum` (migrate.go:276-277) — runner uses ON CONFLICT. So the runner's INSERT no-ops on the duplicate. OK.

Net: this finding is downgraded — the `ON CONFLICT` in the runner saves the day. Leave the finding marked as the latent landmine: any future migration that does its own `INSERT INTO schema_migrations` MUST include `ON CONFLICT` OR the migration body must NOT insert into schema_migrations and rely entirely on the runner.

### Finding 160 — Lifecycle sequencer ignores caller ctx

**Tag:** UNWIRED **Severity:** BLOCKER
**Path:** `internal/infra/lifecycle/sequencer.go:149`

```go
func (m *LifecycleModule) sequencerLoop() {
    ...
    reason := <-m.requestCh
    m.runShutdown(context.Background(), reason)  // <-- background, not the caller's ctx
}
```

`runShutdown` flows `parent` into `racePhase`'s `withDeadline(parent, phaseDeadline)`. With `parent = Background`, the only cancellation source is `phaseDeadline` (a wall-clock timer). The probe-observe-window `select { case <-probeTimer.C: case <-parent.Done(): }` (line 178-182) NEVER takes the `parent.Done()` branch.

**Operator consequence.** `WaitForExit(ctx)` documentation says "blocks until shutdown completes or until the parent ctx fires" (lifecycle.go:272-283). The caller's ctx-fire path returns the in-progress report but doesn't STOP the shutdown — the sequencer continues running on the background ctx. A SIGKILL during force-exit hits in the middle of a shutdown phase that the operator wanted to abort.

### Finding 161 — Production wiring registers ZERO shutdown hooks

**Tag:** UNWIRED **Severity:** BLOCKER
**Path:** `cmd/fhir-subs/wiring.go` (entire), `internal/infra/lifecycle/sequencer.go:258-264`

```bash
$ grep -n "RegisterShutdown\|registerShutdown" cmd/fhir-subs/*.go
(no hits)
```

`buildProductionRuntime` constructs everything but never registers shutdown hooks against `lifecycle.PhaseStopAccepting`, `PhaseDrainInFlight`, or `PhaseCloseConnections`. The lifecycle module's `racePhase` returns immediately for empty phases:

```go
if len(hooks) == 0 {
    dur := m.lctx.Clock().Sub(phaseStart)
    m.lctx.Metrics.Observe(MetricPhaseDurationSeconds, dur.Seconds(), ...)
    return dur, nil, false
}
```

**Operator consequence.** The five-phase shutdown sequence (the backbone of the lifecycle LLD) is decorative. MLLP listener, scheduler, submatcher drain via `pipelineWG`-and-cancel paths in `cmd/fhir-subs/run.go`, NOT via the registered-hook contract. Phase-budgeted draining doesn't happen; the only real budget is `cfg.Lifecycle.ShutdownGracePeriod` enforced by `srv.Shutdown(shutdownCtx)`. The published "five-phase shutdown" architecture is fiction at the wiring level.

```go
    r.Use(d.Auth)
    ...
    r.Get("/metadata", h.getCapabilityStatement)
    ...
})
```

Production wiring (`cmd/fhir-subs/run.go:267-275`) routes "/" to `prod.router` which is the auth-wrapped chi mux. The fall-back `makeMetadata()` at line 268 fires only when `prod == nil`.

**Operator consequence.** A FHIR conformance probe (Inferno, HL7's testkit) calls `GET /metadata` without bearer auth. Production binary returns 401. Audit doc lists S-2.1 "metadata public" as RESOLVED; the production binary disagrees. (Compounds Findings 8 and 111; flagged separately because the previous two were about `RegisterPublicRoutes` not being called — this one is about the route inside `RegisterRoutes` being inside the auth group.)

### Finding 163 — Cross-tenant event leakage via `$events`

**Tag:** INPUT-VALIDATION **Severity:** BLOCKER
**Path:** `internal/api/handlers/pg_stores.go:366-375`

```sql
SELECT id, event_number, topic_url, focus, change_kind, occurred_at,
       resource_change_id
FROM ehr_events
WHERE topic_url = $1
  AND ($2 = 0 OR event_number >= $2)
  AND ($3 = 0 OR event_number <= $3)
ORDER BY event_number ASC
LIMIT 1000
```

No `client_id` predicate. `ehr_events` is the post-fanout topic-matched event log. Two tenants subscribed to `http://example.org/topic/lab-results` see THE SAME ROWS via `$events`. The handler (`subscription_handlers.go:887-965`) checks that the SUBSCRIPTION belongs to the caller's client, but the events list isn't filtered by tenant.

**Operator consequence.** Tenant B calls `GET /Subscription/B-sub-id/$events?eventsSinceNumber=1` and receives the `focus` references (e.g., `Patient/12345`) generated by tenant A's resource changes. PHI cross-tenant disclosure. Multi-tenant deployment is unsafe.

### Finding 164 — Subscriber-controlled retry-pin DoS

**Tag:** INPUT-VALIDATION **Severity:** BLOCKER
**Path:** `internal/channel/resthook/resthook.go:521-537`

```go
func parseRetryAfter(v string) time.Duration {
    ...
    if secs, err := strconv.Atoi(v); err == nil && secs > 0 {
        return time.Duration(secs) * time.Second
    }
    if t, err := http.ParseTime(v); err == nil {
        d := time.Until(t)
        ...
        return d
    }
    return 0
}
```

No upper bound. A subscriber returning `Retry-After: 999999999` parses to `999999999 * time.Second` = ~31 years. Or `Retry-After: Wed, 21 Oct 2099 07:28:00 GMT`. Both flow into `ComputeBackoff` (worker.go:513) and pin `next_attempt_at` for the duration.

**Operator consequence.** A subscriber under attacker control can effectively dead-letter a chosen subset of deliveries. The recovery sweep doesn't help: `recoverStuck` looks for `status = 'delivering'`; the row is `pending` with a far-future `next_attempt_at`. Manual operator intervention is the only escape. The "respect Retry-After" recipe in the channels LLD has no minimum-floor either, so even small persistent bumps drift retry timing far from the operator's RetryConfig curve.

## HIGH findings — narrative

### Finding 166 — Admin reads aren't audited

**Path:** `internal/api/handlers/admin.go` (entire)

Three handlers (`listTopics`, `listSubscriptions`, `listDeadLetters`) each return PHI-adjacent data. None call `a.deps.Audit.Append`. The package comment promises "the shared secret is the gate; network-level scoping is the operator's responsibility" — but doesn't address recording who/what was read. SOX, HIPAA, and most compliance frameworks require audit trail for privileged-bypass-of-OAuth read paths.

### Finding 169-170 — HL7 MSH-7 parser is dialect-fragile

`messageDateTime` (processor.go:935-984) parses MSH-7 with case-sensitive segment-ID match (`first[0] != 'M'`) and naïve field separator walking (no escape handling). Vendor dialects with lowercase `msh` or escaped field separators silently fall back to wall-clock `Now()`. The fallback is silent; no metric, no log. EHR-stamped time differs from on-disk `occurred_at` for entire sites.

### Finding 175-176 — Logging redactor is structurally incomplete

The redactor strips fields by NAME match (`PHIFieldNames`) and partial URLs by `QueryStringTargets`. It does NOT:
- redact PHI inside untyped `value` strings
- canonicalize key names (so a non-snake_case variant like `mrnNumber` slips through)
- strip path segments from URL fields (only `?...`)

A caller logging `slog.Info("processed", "mrnNumber", "12345")` defeats the redactor entirely. The package's docstring promises "PHI fields are dropped at info+" — the implementation is whitelist-on-key-name-only.

### Finding 178-179 — Scheduler retry-attempt accounting races

`recoverStuck` uses an unlocked `UPDATE ... SET attempts = attempts + 1` query. Two pods running the recovery sweep simultaneously double-increment `attempts`. The retry budget exhausts twice as fast on multi-pod deployments. The advisory_lock pattern was used for migrations (Finding B-33 RESOLVED) but not extended to the scheduler.

### Finding 181-182 — Orphan documentation pages

`docs/admin-tool.md`, `docs/admin-tool-review.md`, `docs/adapter-authoring-guide.md`, `docs/adapter-conformance-checklist.md` exist on disk but aren't in `mkdocs.yml`'s `nav:`. mkdocs builds them (no `--strict`); search indexes them. Operators searching for "admin tool" find a page that describes an unwired surface (Finding 7). The orphan-page pattern means "documented but unmaintained" is structurally indistinguishable from "documented and supported."

## MEDIUM findings — narrative

### Finding 187 — Migration crash recovery hole

`pg_advisory_lock` is session-scoped. On the holder's connection crash, Postgres releases the lock. But the partial DDL inside the migration's transaction has either committed (everything before the failed statement) or rolled back (entire transaction). Multi-statement migrations that ran several `CREATE TABLE` before a syntax error commit some, abort the rest — and `schema_migrations` doesn't get the version row (because the runner's `INSERT INTO schema_migrations` is in the same transaction and rolled back too).

The next pod's runner sees: schema_migrations has ('0001','...') but not ('0002', ...). It tries to apply 0002. 0002's body contains `CREATE TABLE foo` — but `foo` already exists from the partially-committed prior run. The migration aborts with `relation "foo" already exists`. The advisory lock didn't help; what the runner needs is per-migration idempotency (every CREATE statement uses `IF NOT EXISTS`). Some migrations do, some don't.

### Finding 196 — Scheduler tunables unsurfaced

`RecoveryInterval`, `StuckThreshold`, `DispatchConcurrency` defaults (30s, 5m, 1) are reasonable for low-volume installs and bad for high-volume. With 1000 deliveries/min and DispatchConcurrency=1, deliveries serialize on rest-hook latency. There's no operator knob.

## LOW findings — narrative

### Finding 197 — Non-strict YAML decoding

The audit-supplement-1 caught the `${env:...}` placeholder issue. The orthogonal issue: a typo in a key name (e.g., `databse.url:`) is silently absorbed by `Extra` via `yaml:",inline"`. `yaml.v3`'s `Decoder.KnownFields(true)` would surface this; it's not enabled. Compounds the validation gap.

### Finding 199 — URL path-segment leakage

`QueryStringTargets` strips only the `?query` portion of `endpoint` / `url` / `webhook_url` / `callback_url` / `href`. A path-segmented URL `/api/webhooks/<patient-uuid>/inbound` keeps the patient UUID intact in logs. The redactor's design assumes secrets are only in query parameters; in practice REST APIs frequently encode IDs in path segments.

### Top three most-embarrassing new items

1. **Finding 163 — `$events` cross-tenant PHI disclosure.** `ehr_events` is a topic-shared event log; the handler validates subscription ownership but the underlying query selects every event on the topic, regardless of tenant. Two tenants subscribed to the same topic see each other's `focus` references (which encode patient IDs). 2. **Finding 161 — The five-phase shutdown is fiction.** `cmd/fhir-subs/wiring.go` registers ZERO shutdown hooks against the lifecycle module's three operational phases. `racePhase` returns instantly for empty phases. The lifecycle module is a probe handler; the rest is dead.
